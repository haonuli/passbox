# CLAUDE.md - 项目开发规范与工作流指南

> **重要提示**：本文档是本项目开发的最高行为准则。AI 助手或开发者在执行任何任务前，必须严格阅读并遵守以下规则。严禁跳过任何流程或降低交付标准。

## 🎯 核心原则 (Core Principles)

1. **状态实时同步**：每完成一个任务（或任务的一个阶段），必须立即更新 [TASK_TRACKING.md](./TASK_TRACKING.md) 中的【任务状态追踪】表格，确保进度透明、可追溯。
2. **严格质量门禁**：每完成一个任务，必须进行充分的自测，**确保 100% 符合交付标准后**，方可开始下一个任务。严禁“带病推进”或“先开发后补文档”。
3. **原子化 Git 提交**：每完成一个独立任务并测试通过后，必须立即执行 Git 提交（Commit）和推送（Push）。保持提交记录的原子性和清晰度。
4. **标准七步工作流**：面对**任何中大型新需求**，必须严格按照以下顺序执行，不可颠倒或跳过任一环节：
   `需求定义` ➔ `技术方案` ➔ `任务拆分与排期` ➔ `测试策略` ➔ `开发` ➔ `测试` ➔ `评审合并`

---

## 🔄 标准开发与交付工作流 (Standard Workflow)

### 0️⃣ 前置动作：任务分级 (Task Triage)
在开始任何工作前，先评估任务规模，选择对应的执行路径：
- 🟢 **微小任务 (Bug修复/文案修改/UI微调)**：
  `明确问题` ➔ `开发修复` ➔ `自测验证` ➔ `Git提交`
- 🟡 **中大型需求 (新功能/架构调整/复杂重构)**：
  必须严格执行下方的【完整七步规范流程】。

---

### 🟡 完整七步规范流程 (针对中大型需求)

#### 阶段 1：需求定义与验收标准 (Requirements & AC)
- **目标**：明确做什么，以及怎么做才算成功。
- **产出**：需求背景、核心功能点，**必须包含明确的验收标准 (Acceptance Criteria, AC)**。
- **调用**：@资深产品经理
#### 阶段 2：技术方案设计 (Technical Design)
- **目标**：明确怎么做，规避技术风险。
- **产出**：模块设计、API 接口定义、数据库变更说明。
- ⚠️ **规则**：如果在后续开发中发现技术方案有误，**必须立即返回此阶段更新文档**，严禁文档与代码脱节。

#### 阶段 3：任务拆分与开发排期 (Task Breakdown & Estimation) 🌟 核心控盘点
- **目标**：将复杂需求拆解为可独立执行、验证的原子任务，化繁为简。
- **产出**：
  1. **子任务清单**：按执行顺序列出具体任务（如：建表 ➔ 写后端接口 ➔ 写前端页面 ➔ 联调）。
  2. **依赖关系**：明确哪些任务必须前置完成。
  3. **预估耗时**：为每个子任务提供预估时间。
- ⚠️ **AI 执行要求**：AI 必须严格按照拆分后的清单顺序逐个执行，**禁止一次性生成所有代码**。

#### 阶段 4：测试策略与 Checklist (Test Strategy)
- **目标**：明确如何验证各个子任务及整体功能。
- **产出**：基于阶段 3 的子任务，产出**核心验收 Checklist** 或直接编写**自动化测试代码骨架 (TDD)**。

#### 阶段 5：开发与单元测试 (Dev & Unit Test)
- **目标**：高质量实现子任务。
- **要求**：严格遵循技术文档，编写清晰的代码，并确保每个子任务的**单元测试通过**。

#### 阶段 6：集成测试与验收 (QA & Acceptance)
- **目标**：验证是否满足阶段 1 的 AC。
- **要求**：执行阶段 4 的 Checklist 或自动化测试。如有 Bug，修复后重新验证，直到 100% Pass。

#### 阶段 7：评审、合并与状态更新 (Review & Close)
- **要求**：
  1. 发起 Pull Request / Merge Request (如适用)。
  2. 确保所有代码已推送到远程仓库。
  3. **强制更新**：更新 [TASK_TRACKING.md](./TASK_TRACKING.md) 中的【任务状态追踪】表格，标记主任务及子任务的状态。

## 📦 Git 提交规范 (Git Commit Guidelines)

每个任务/子任务完成后，必须使用清晰的 Commit Message 提交代码：
- `feat:` 新增功能
- `fix:` 修复 Bug
- `docs:` 文档更新 (需求/技术/测试/排期文档)
- `test:` 增加或修改测试用例
- `refactor:` 代码重构（不影响功能）

*示例*：`feat(auth): 完成用户登录接口开发 (子任务 3.2)`

## 🤖 AI 助手执行指令 (AI Execution Prompt)

如果你是读取此文档的 AI 助手，请在每次收到新需求时，按以下逻辑响应：
1. **确认流程**：明确告知用户当前处于“七步工作流”的哪一步。
2. **拒绝越级**：如果用户直接要求写代码，请礼貌提醒：“根据 CLAUDE.md，我们需要先进行需求定义和技术方案，然后再做任务拆分，请问现在从阶段 1 开始吗？”
3. **逐步执行**：在【阶段 3】产出任务拆分清单后，**必须暂停**，等待用户确认清单合理后，再开始逐个执行子任务。
4. **更新状态**：在每次完成一个子任务或阶段后，提供更新后的【任务状态追踪】表格 Markdown 代码。
